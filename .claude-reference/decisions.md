# Draplo — Decisions & Gotchas

## Key Decisions

### Why Claude Sonnet 4.6 and not Haiku or Opus?
**Decision:** Use Sonnet 4.6 for all generation tasks
**Reason:** Haiku is too weak for generating complex, consistent architectural documents — it hallucinates model names and produces inconsistent schemas. Opus is overkill and 5x the cost. Sonnet 4.6 at $3/$15 per MTok hits the sweet spot: smart enough for architecture generation, cheap enough at $0.15-0.25 per generation.
**Trade-off:** Slightly higher cost than Haiku, but dramatically better output quality.

### Why XML tags for output parsing instead of JSON?
**Decision:** AI wraps each file in `<file path="...">...</file>` XML tags
**Reason:** JSON output is fragile — one misplaced quote in a PHP migration string breaks the entire parse. XML tags are robust because the content between tags can contain any characters including JSON, PHP, markdown. Also, Claude is better at producing well-formed XML tags than perfectly escaped JSON with code inside.
**Trade-off:** Custom parser needed, but it's trivial (single regex).

### Why GitHub OAuth and not email/password auth?
**Decision:** GitHub is the only login method
**Reason:** Our target users are developers who all have GitHub accounts. GitHub OAuth gives us repo creation permissions in the same flow — no separate "connect GitHub" step. Reduces friction and eliminates password management.
**Trade-off:** Non-GitHub users (GitLab, Bitbucket) are excluded for now. Add later if demand exists.

### Why BYOS (Bring Your Own Server) and not managed hosting?
**Decision:** Users provide their own server, Draplo automates provisioning and deployment
**Reason:** Managed hosting means we host user apps on our server — legal liability for user data (GDPR), infrastructure costs that scale with users (kills margins), and operational burden (monitoring 50+ apps 24/7). BYOS means zero infrastructure cost for us, zero legal liability, and users get full ownership of their server. If they cancel Draplo, their apps keep running. This is the strongest "zero vendor lock-in" pitch possible.
**Trade-off:** More complex first-deploy UX (user needs to enter API key, wait for server provisioning). But target users are developers — they appreciate server ownership.

### Why Coolify on user's server and not raw Docker?
**Decision:** Install Coolify on user's VPS, manage via Coolify API
**Reason:** Coolify provides a UI for the user (they can manage their server visually after Draplo sets it up), handles SSL via Let's Encrypt, supports auto-deploy on git push, and has a comprehensive REST API. Raw Docker deploy would require us to build all of that ourselves.
**Trade-off:** Dependency on Coolify project continuing development. But it's open source — worst case, the installed version keeps working.

### Why queue AI generation instead of synchronous?
**Decision:** Generation runs as a queued job (GenerateProjectJob)
**Reason:** Claude API calls take 15-30 seconds for 15K output tokens. Synchronous HTTP requests would time out. Queued jobs let us show a "Generating..." progress UI while the job runs in the background, and retry on API failures.
**Trade-off:** More complex UX (polling for completion) but much more reliable.

### Why Laravel 12 for the skeleton and not 13?
**Decision:** Generated projects use Laravel 12
**Reason:** Laravel 13 is new (2026), and package compatibility (especially stancl/tenancy) is not guaranteed. Laravel 12 is stable with full ecosystem support. Developers can upgrade themselves later. We'll add Laravel 13 as a skeleton option once the ecosystem catches up.
**Trade-off:** Miss out on Laravel 13's AI SDK, but that's a feature users can add themselves.

## Gotchas

### "The Monolithic Architect" design system
**Decision:** Use a custom design system with "Technical Brutalism" aesthetic, Material Symbols icons, Berkeley Mono monospace, and no standard emoji
**Reason:** The product must not look like it was generated by AI. Standard emoji, generic Tailwind UI, and purple-gradient-on-white instantly signal "AI slop." The Monolithic Architect system uses tonal layering instead of borders, glass panels, and editorial typography to create a premium feel that developers trust. Material Symbols are consistent and professional. Berkeley Mono for data reinforces the developer-tool identity.
**Trade-off:** Higher implementation effort than using a component library like shadcn. But the differentiation is worth it.

### Three.js hero on landing page
**Decision:** Interactive 3D element on landing page instead of static images or screenshots
**Reason:** Every AI tool landing page in 2025-2026 looks identical — dark background, gradient text, feature grid. A Three.js 3D visualization of "architecture being assembled" is memorable, demonstrates technical capability, and creates an emotional "wow" moment. It signals that Draplo is not a template — it's engineered.
**Trade-off:** Performance cost (must maintain 60fps), fallback needed for low-end devices, extra bundle size (~150KB for Three.js). Use dynamic import and intersection observer to only load when hero is visible.

### Server provider API key security
**Decision:** Encrypt API keys at rest with Laravel's encrypt(), decrypt only during active deploy operations
**Reason:** BYOS means we temporarily hold user's Hetzner/DO/Linode/Vultr API keys. These keys can provision servers and incur costs. They MUST be encrypted at rest and never logged, never exposed in UI after initial entry, and never included in any debug output.
**Trade-off:** Keys must be decrypted for each deploy operation, but the security benefit is non-negotiable.
**Decision:** Use AGPL-3.0 for open source release
**Reason:** AGPL requires anyone who hosts a modified version as a public service to publish their source code. This prevents "fork → rebrand → sell as SaaS" without contributing back. Self-hosting for personal or internal use is unrestricted. This is the same model used by GitLab, Grafana, and MinIO.
**Trade-off:** Some companies avoid AGPL software due to legal caution. But our target (indie developers, small teams) doesn't care about this.

### Feature flags for flexible deployment
**Decision:** All major features (Coolify, GitHub, templates) are behind .env feature flags
**Reason:** Self-hosters may not have Coolify (generate-only mode), or may prefer ZIP over GitHub. Feature flags let the same codebase serve any self-hosted instance without code forks.
**Trade-off:** More conditional logic in controllers and views, but worth the flexibility.

## Gotchas

### Prompt caching requires consistent system prompt
Anthropic's prompt caching works by matching the exact same system prompt bytes. If the system prompt changes (even whitespace), the cache is invalidated. Keep the system prompt in a versioned file (`app/Prompts/system-prompt.md`) and don't modify it per-request. All customization goes in the user message, not the system prompt.

### GitHub API rate limits for repo creation
Creating a repo and pushing files via the Git Data API involves multiple requests: create repo → create blobs → create tree → create commit → update ref. For a project with 30+ files, that's 35+ API calls. GitHub allows 5000 authenticated requests/hour, so we're fine, but implement retry with exponential backoff for 403/429 responses.

### Coolify auto-deploy webhook conflicts
When we push to GitHub AND Coolify has auto-deploy enabled, Coolify will try to deploy immediately. If the push includes migration files, the deploy might fail if the database hasn't been seeded yet. Solution: disable auto-deploy for the initial push, run migrations manually via Coolify API, then enable auto-deploy for subsequent pushes.

### Generated migrations must be ordered correctly
When AI generates multiple migration files, foreign key constraints require correct ordering (create `users` before `posts` that references `users`). The system prompt must explicitly instruct: "Generate migrations in dependency order. Tables with no foreign keys first."

### Wizard data must be validated before generation
Garbage in = garbage out. If wizard data has empty model names, duplicate field names, or contradictory settings (multi-tenant: false but role: tenant_admin), the AI will produce inconsistent output. Validate wizard data thoroughly before sending to AI.

### ZIP download must exclude .git directory
When generating ZIP for download, never include `.git/` — it would contain our skeleton's git history. Generate a clean ZIP with only the project files.
